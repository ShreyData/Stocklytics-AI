AI Coding Instructions

- Operational database: Firestore
- Analytics database: BigQuery
- AI provider: Gemini API
- Auth: Firebase Auth

How to behave
- Act like a senior engineer working inside an existing planned system.
- Follow the documented architecture and contracts exactly.
- Do not invent new modules, new major services, or new platform components unless explicitly requested.
- Keep solutions simple, production-aware, and easy for a small team to maintain.
- Prefer clear service logic over clever abstractions.

Module boundary rules
- Preserve these module names exactly:
  - Inventory Module
  - Billing Module
  - Customer Module
  - Analytics Module
  - Alerts Module
  - AI Module
  - Data Pipeline Module
  - API Module
  - Frontend Module
- Keep module-specific code inside its own module folder.
- Put only shared utilities in common or shared folders.
- Do not tightly couple one module to another module's internal repository code.
- Use clean service interfaces between modules inside the monolith.

Naming rules
- API paths: lowercase kebab-case
- JSON fields: snake_case
- Firestore collections: plural snake_case
- BigQuery datasets and tables: snake_case
- Python files, functions, and variables: snake_case
- Python classes: PascalCase
- Enum and status constants: UPPER_SNAKE_CASE where appropriate
- Use these field names exactly:
  - store_id
  - product_id
  - transaction_id
  - customer_id
  - alert_id
  - chat_session_id
  - pipeline_run_id
  - idempotency_key
  - request_id
  - analytics_last_updated_at
  - freshness_status

Critical business rules
- Billing must be strictly atomic.
- If any billing item has insufficient stock, the entire billing request fails.
- No partial stock deduction is allowed.
- Billing create requests must support idempotency_key.
- Same idempotency_key plus same payload must return the original result without duplicate transaction creation.
- Same idempotency_key plus different payload must return a conflict error.
- Inventory is the live source of truth for stock.
- Stock must never go negative.
- Analytics responses must always include analytics_last_updated_at and freshness_status.
- Alerts must use lifecycle states:
  - ACTIVE
  - ACKNOWLEDGED
  - RESOLVED
- AI must use structured system data only.
- Do not add heavy RAG or vector database behavior.
- Data pipeline retries up to 3 times and must preserve failure and recovery behavior.

AI module rules
- AI context must come only from:
  - analytics summary
  - alerts
  - inventory snapshot
- AI responses must be grounded in available system data.
- If analytics data is stale or delayed, the response must mention freshness clearly.
- Do not invent unsupported trends, reasons, or recommendations.

API rules
- Base path is /api/v1
- Use JSON request and response bodies
- Use the approved request and response field names exactly
- Use consistent error format:
  - request_id
  - error.code
  - error.message
  - error.details
- Do not change endpoint names or response shapes without explicit approval

Database rules
- Firestore is for operational data
- BigQuery is for analytics and summary queries
- Keep store_id on all major records
- Do not move analytics computation into live operational request paths
- Do not duplicate detailed transaction history into customer documents

Code structure rules
- Keep routers thin
- Put business logic in services
- Put storage access in repository or data access layers if the codebase uses them
- Keep validation explicit
- Keep helper functions small and readable
- Avoid overengineering and avoid premature abstraction

Frontend rules
- Frontend should call backend APIs only
- Frontend should not call Firestore or BigQuery directly
- Billing UI should generate and reuse idempotency_key for safe retries
- Frontend must display analytics freshness
- Frontend must support alert lifecycle actions clearly

What to avoid
- Do not rename modules
- Do not introduce microservices
- Do not introduce event-driven architecture unless explicitly requested
- Do not add vector databases, embeddings pipelines, or RAG systems
- Do not change naming conventions
- Do not change business rules silently
- Do not mix camelCase and snake_case in API payloads

Before generating code
- Identify the assigned module or branch first
- Read the relevant implementation guide in implimentation/
- Read the related planning docs
- Confirm the API contract, database fields, and business rules that apply
- Stay within the assigned module unless shared changes are necessary

When generating code
- Match the approved API contract exactly
- Match the approved database design exactly
- Include validation and failure handling
- Keep logs meaningful and include important IDs where relevant
- Add comments only where they truly help explain non-obvious logic
- Keep code easy for another teammate to continue

Testing expectations
- Add at least one happy path test
- Add key validation tests
- Add key failure-path tests
- For billing, always cover atomic failure and idempotency behavior
- For alerts, cover lifecycle transitions
- For analytics, cover freshness fields
- For AI, cover grounded response behavior

PR and collaboration rules
- Keep changes focused to one module or one clear task
- Do not make unrelated refactors in the same change
- If changing a shared contract, update docs first or in the same PR
- Keep code consistent with existing project decisions
- Prefer small, reviewable commits and PRs

Output style for AI-generated code help
- Be practical
- Be concise
- Prefer implementation-ready answers
- If there is a conflict between your idea and the project docs, follow the project docs
- If a rule is unclear, prefer the simpler option that preserves modularity and consistency
